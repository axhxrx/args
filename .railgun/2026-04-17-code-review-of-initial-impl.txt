The tokenizer currently accepts a couple of malformed option forms that silently change the parsed result instead of surfacing an error. Those cases affect the core parsing contract, so the patch is not fully correct yet.

     /Users/mason/Documents/CODE/axhxrx-megarepo/args/src/tokenizer.ts:74-79
     This branch always consumes the next argv element as the value for a required option. [P2] Reject undocumented `-xVALUE` short-option syntax
     /Users/mason/Documents/CODE/axhxrx-megarepo/args/src/tokenizer.ts:108-112
     `rest.length > 0` makes any suffix after a value-taking short option parse successfully, so undocumented forms like `-n3` are accepted and bundle-looking inputs like `-ab` become indistinguishable from `-a` with value `b`. Covered by `attached-without-equals short value (e.g. -n3) throws`, `attached-without-equals path-like short value throws`, and `short flag given an attached value throws`.
