The tokenizer currently accepts a couple of malformed option forms that silently change the parsed result instead of surfacing an error. Those cases affect the core parsing contract, so the patch is not fully correct yet.

[P2] Reject option-shaped tokens as missing values
     /Users/mason/Documents/CODE/axhxrx-megarepo/args/src/tokenizer.ts:74-79
     This branch always consumes the next argv element as the value for a required option. In inputs like `['pos', '--retry', '--verbose']` (and the analogous short form below), that silently swallows the next option instead of raising `Missing value`, so registered or unknown options disappear and the grouping changes. `SPEC.md` documents option-shaped next tokens as missing values, so treating them as ordinary values here breaks the advertised contract.

[P2] Reject undocumented `-xVALUE` short-option syntax
     /Users/mason/Documents/CODE/axhxrx-megarepo/args/src/tokenizer.ts:108-112
     `rest.length > 0` makes any suffix after a value-taking short option parse successfully, so undocumented forms like `-n3` are accepted and bundle-looking inputs like `-ab` become indistinguishable from `-a` with value `b`. That means unsupported short-option bundles are silently reinterpreted instead of rejected, and a real `-b` option is never seen. The public contract only allows `-x VALUE` or `-x=VALUE`, so this branch should not accept bare attached suffixes.
